**YAML Frontmatter** (per D-13):
```yaml
---
name: philatex-update
description: |
  Update the SAPA website with content from a new Philatex newsletter PDF.
  Reads the PDF, checks for duplicate editions, then hands off to the
  extraction and update workflow. Use when asked to "update newsletter",
  "process Philatex", "update site with new Philatex", or given a PDF path
  with "/philatex-update".
allowed-tools:
  - Read
  - Write
  - Edit
  - Bash
  - Grep
  - Glob
---
```

**Skill Body** — use the simple standalone pattern with numbered steps (per D-14). The body must contain these sections:

**1. Input Parsing** (per D-13, Pattern 6 from FEATURES.md):
- Parse the first argument as the PDF path (required)
- If no PDF path given, ask the user to provide one before proceeding
- Example: `/philatex-update ~/Downloads/philatex-q3-2026.pdf`

**2. File Validation** (per D-10 step 2):
- Validate the file exists using `[ -f "$PDF_PATH" ]`
- Validate it has a `.pdf` extension
- If invalid, report the error and stop

**3. Edition Identification** (per D-04, D-06):
- Read the PDF using the Read tool (no `pages` parameter needed — newsletters are under 10 pages)
- Identify the edition from page 1 masthead: "THE PHILATEX" with quarter and year
- Construct the edition ID in `YYYY-QN` format (e.g., "2026-Q3")
- Map quarter names: First=Q1, Second=Q2, Third=Q3, Fourth=Q4
- Include instructions: "The newsletter always identifies itself on page 1. Do NOT rely on the filename."

**4. Duplicate Check** (per D-05):
- Read `data/newsletters/newsletters.json`
- Check the `id` field of every entry in the `newsletters` array
- If a match is found: report "Edition {ID} already exists in newsletters.json. This newsletter has already been processed. No changes made." and STOP
- If no match: proceed to step 5

**5. Handoff to Phase 11 Workflow** (per D-10 step 6, D-11):
- State clearly: "Edition {ID} is new. Proceeding with extraction and update workflow."
- Pass the PDF path and identified edition (year, quarter, ID) as context to the Phase 11 agent
- Include a placeholder for the Phase 11 agent reference: "TODO: Phase 11 will define the agent at `.claude/agents/philatex-newsletter-agent.md`"
- Note per D-12: "The skill resumes after the agent completes to handle the human checkpoint (Phase 12). The commit happens in the skill after checkpoint approval, not in the agent."

**6. Reference Checklist** (per D-07, D-08, D-09):
Create an inline reference checklist with four subsections. This is a quick-reference card, NOT a duplicate of the schemas.

6a. **JSON Data Files:**
- `data/newsletters/newsletters.json` — key fields: `id` (YYYY-QN), `title`, `quarter` (First/Second/Third/Fourth), `year`, `publishDate` (YYYY-MM-DD, first day of quarter), `filePath` (public/SAPA-PHILATEX-...), `description`, `featuredArticles[]`, `highlights[]`, `tags[]`, `pageCount`, `fileSize`, `status`. Validate against `data/schemas/newsletter.schema.json`. Note: `featuredArticles.category` enum does not include "Calendar" or "Humor" — flag invalid categories for human review.
- `data/meetings/meetings.json` — key fields: `id` (YYYY-MM-DD), `date`, `time` (doorsOpen required), `location`, `type` (regular/business/auction/exhibition/social/special/picnic/holiday), `title`, `presenter`, `cancelled`. Validate against `data/schemas/meeting.schema.json`. Note: existing data uses `bogStart` as an extension not in schema — include for BOG meetings.
- `metadata` block in newsletters.json — update: `lastUpdated`, `totalIssues` (+1), `latestIssue` (new ID)

6b. **HTML Pages — Section IDs receiving updates:**
- `index.html` — `#main-content` (upcoming meeting info, latest newsletter link), `#meeting-schedule` (schedule highlights)
- `newsletter.html` — `#main-content` (current issue section, archive year section)
- `meetings.html` — `#main-content`, `#meeting-schedule-container` (meeting list), `#calendar-container` (calendar widget data)
- `about.html` — `#main-content` (Board of Governors roster, if changed)
- `contact.html` — mailing address block (if changed)

6c. **ICS Naming Conventions:**
- Individual meeting files: `data/calendar/YYYY-MM-DD-meeting.ics` (e.g., `2026-07-03-meeting.ics`)
- Quarterly schedule bundle: `public/sapa-qN-YYYY-meetings.ics` (e.g., `public/sapa-q3-2026-meetings.ics`)
- ICS UID format: `YYYYMMDDTHHMMSSZ-sapa@sastamps.org`
- ICS PRODID: `-//San Antonio Philatelic Association//SAPA Meetings//EN`

6d. **PDF Naming Convention:**
- New PDF goes to: `public/SAPA-PHILATEX-[Quarter]-Quarter-[Year].pdf`
- Example: `public/SAPA-PHILATEX-Third-Quarter-2026.pdf`
- Quarter word mapping: Q1=First, Q2=Second, Q3=Third, Q4=Fourth

**7. Permitted-File Scope Boundary** (per D-01, D-02, D-03):
Enumerate the EXACT list of files the agent may modify without human confirmation:

```
PERMITTED FILES (agent may modify freely):
  data/newsletters/newsletters.json
  data/meetings/meetings.json
  data/calendar/*.ics              (new files only)
  public/*.ics                     (new files only)
  public/*.pdf                     (additions only, no deletions)
  index.html
  newsletter.html
  meetings.html
  about.html
  contact.html

SCOPE RULE: Any file NOT on this list requires explicit human
confirmation before modification. If the agent identifies out-of-scope
changes it believes are needed, it must report them in the checkpoint
summary rather than making them.
```

**Important Rules section** at the end:
- The skill does NOT extract content, update files, validate schemas, or run build commands — those are Phase 11 agent responsibilities (per D-11)
- The skill does NOT commit changes — that happens after the Phase 12 checkpoint (per D-12)
- The skill does NOT ask the user what edition the PDF is — it reads the PDF to determine this (per D-04)
