**YAML Frontmatter:**
```yaml
name: philatex-newsletter-agent
description: Reads a Philatex newsletter PDF, extracts structured content, validates against schemas, generates a proofreading report, and updates all SAPA website data files. Spawned by /philatex-update skill.
```

**Body structure using `<role>` block + numbered steps:**

The agent receives from SKILL.md Step 5:
- `PDF_PATH`: absolute path to the newsletter PDF
- `EDITION_YEAR`: e.g., 2026
- `QUARTER_NAME`: e.g., "Third"
- `EDITION_ID`: e.g., "2026-Q3"

**Step 1: PDF Intake and Structure Survey**
- Read the PDF using the Read tool (no `pages` parameter needed -- newsletters are under 10 pages)
- Before extracting, describe the newsletter's structure: how many pages, what sections are present on each page. This surfaces format changes early (per Pitfall 7 prevention).
- Get file size via `stat -f "%z" "$PDF_PATH"` and convert to human-readable (bytes -> KB/MB)
- Count pages from the Read tool output

**Step 2: Proofreading Report (CONT-02)**
- Review the newsletter for: typos, grammatical errors, layout/formatting issues, factual inconsistencies within the document (per D-05)
- When discrepancies between PDF sections affect extraction (e.g., prose says "April" but calendar shows May 29 for a program), note them prominently (per D-06)
- Save report to `.planning/reviews/YYYY-QN-newsletter-review.md` (per D-07). Example path: `.planning/reviews/2026-Q3-newsletter-review.md`
- Create `.planning/reviews/` directory if it does not exist
- The report format should follow the precedent set by quick task #2 (`.planning/quick/2-review-sapa-philatex-q2-2026-newsletter-/`)

**Step 3: Structured Content Extraction (CONT-01)**
- Perform a single-pass extraction against the PDF content already in context. Extract into a structured mental model (held in conversation memory, not a temp file):

```
Extraction targets:
- newsletter_metadata: title, quarter, year, publishDate, description, featuredArticles[], highlights[], tags[]
- meetings: array of meeting objects matching meetings.json format (13 per quarter typical)
- officers: key-value pairs of role:name (for about.html comparison)
- announcements: list of discrete announcements (new members, condolences, events)
- tsda_shows: stamp show entries (dates, names, locations)
```

**Meeting extraction rules** (critical -- source of most errors):
- Source meeting DATES from the calendar TABLE, not from prose sections. The calendar is a 3-column table (one column per month). Tell yourself: "Column 1 = Month 1, Column 2 = Month 2, Column 3 = Month 3."
- If prose contradicts the calendar table on a date, trust the calendar table and flag the discrepancy in the proofreading report (per D-06).
- Do NOT apply default times blindly. The picnic has a non-standard start time (6:00 PM instead of 6:30 PM). Check each meeting individually.
- Meeting type mapping from newsletter text:
  - "NO MEETING" / "HOLIDAY" -> type: "holiday", cancelled: true
  - "Board Of Governors" / "BOG" -> type: "business"
  - "Club Auction" -> type: "auction"
  - "Bourse" -> type: "social"
  - "Stamp Program" (with presenter) -> type: "regular"
  - "Annual Club Picnic" -> type: "picnic"
  - Anything unclear -> type: "regular"
- Meeting ID = date in YYYY-MM-DD format
- Include `bogStart: "7:15 PM"` for BOG/business meetings (matches existing data pattern, even though not in schema)
- Cancelled/holiday entries: `time: { doorsOpen: "N/A", meetingStart: "N/A", meetingEnd: "N/A" }`, `cancelled: true`

**[UNVERIFIED] marking rules (CONT-04, per D-01 and D-02):**
- For free-text fields (description, specialNotes, highlights): place `[UNVERIFIED]` inline alongside the best-guess value. Example: `"description": "[UNVERIFIED] Stamp Program by Joe Perez -- newsletter prose says April but calendar shows May 29"`
- For schema-constrained fields (enums, dates): use the best-guess valid value and log uncertainty in a nearby free-text field. Example: `specialNotes: ["[UNVERIFIED] Meeting type inferred as 'auction' -- newsletter text was ambiguous"]`
- Any extraction where confidence is below high should be marked

**Step 4: Schema Validation (CONT-03)**
- Before writing any data file, validate extracted data against the schemas:
  - Read `data/schemas/newsletter.schema.json` and verify the new newsletter entry has all required fields, correct types, valid enum values, and correct patterns (id matches `^[0-9]{4}-Q[1-4]$`, publishDate is valid date, filePath matches `^public/.*\.pdf$`)
  - Read `data/schemas/meeting.schema.json` and verify each meeting entry has all required fields (id, date, time with doorsOpen, location with name, type from valid enum)
  - Note: the time object schema has `additionalProperties: false` but existing data includes `bogStart`. Include `bogStart` for BOG meetings to match existing data (the schema will flag this if validated programmatically -- note this as a known schema gap, do not treat it as an error)
- If any required field is missing or an enum value is invalid: flag the violation, do NOT silently accept it, and attempt to fix it. If unfixable, mark as [UNVERIFIED] and continue.
- Report all validation findings (passes and failures) in the extraction output.

**Step 5: Data File Updates (WORK-03 part 1)**
- **newsletters.json**: Read the current file. Prepend the new entry at the top of the `newsletters` array (most recent first). Update the `metadata` block: set `lastUpdated` to the newsletter's `publishDate` value formatted as ISO 8601 UTC (`YYYY-MM-DDT00:00:00.000Z`), increment `totalIssues` by 1, set `latestIssue` to the new edition ID. Use the Write tool to write the complete updated file.
- **meetings.json**: Read the current file. Identify the last date present. Append only new entries (dates after the last existing date) at the end of the `meetings` array (chronological order, oldest first). Use the Write tool to write the complete updated file.

**Step 6: ICS Calendar File Generation (WORK-03 part 2, per D-10 through D-13)**

Generate BOTH individual per-meeting ICS files AND a quarterly aggregate ICS file.

**Individual ICS files** (one per meeting in `data/calendar/`):
- Filename: `YYYY-MM-DD-meeting.ics` (or `-picnic.ics`, `-holiday.ics` as appropriate per D-11)
- Use UTC timestamps for DTSTART/DTEND (e.g., `DTSTART:20260710T003000Z` for 6:30 PM CDT)
- UID format: `YYYYMMDDTHHMMSSZ-sapa@sastamps.org`
- DTSTAMP: first day of quarter at midnight UTC (e.g., `20260701T000000Z`)
- PRODID: `-//San Antonio Philatelic Association//SAPA Meetings//EN`
- CALSCALE: GREGORIAN
- Cancelled meetings: STATUS:CANCELLED, 1-minute duration, LOCATION: "Meeting Cancelled"
- Active meetings: STATUS:CONFIRMED, LOCATION uses `\,` escaping for commas
- Standard location: `MacArthur Park Lutheran Church\, Building 1\, 2903 Nacogdoches Road\, San Antonio\, TX 78217`
- CATEGORIES appropriate to meeting type

Use existing ICS files as exact format templates:
- Regular meeting: model after `data/calendar/2026-04-10-meeting.ics`
- Holiday/cancelled: model after `data/calendar/2026-04-03-meeting.ics`
- Picnic/special: model after `data/calendar/2026-06-19-picnic.ics`

**Quarterly aggregate ICS file** (one file in `public/`):
- Filename: `sapa-qN-YYYY-meetings.ics` (e.g., `public/sapa-q3-2026-meetings.ics`) per D-12
- Use LOCAL timestamps for DTSTART/DTEND (no TZID prefix, no Z suffix)
- UID format: `sapa-YYYY-MM-DD@sastamps.org`
- Include X-WR-CALNAME, X-WR-TIMEZONE (America/Chicago), X-WR-CALDESC headers
- PRODID: `-//San Antonio Philatelic Association//SAPA Meeting Calendar//EN` (note: different from individual files)
- Contains one VEVENT per meeting in the quarter
- Model after `public/sapa-q2-2026-meetings.ics`

**Time conversion reference (CDT = UTC-5):**
- 6:00 PM CDT = 23:00 UTC (same day) -- picnic
- 6:30 PM CDT = 23:30 UTC (same day) -- doors open / cancelled placeholder
- 7:15 PM CDT = 00:15 UTC (next day)
- 7:30 PM CDT = 00:30 UTC (next day)
- 9:00 PM CDT = 02:00 UTC (next day)

**IMPORTANT timezone note:** Check the quarter's dates to determine if CDT (UTC-5) or CST (UTC-6) applies. CDT runs from second Sunday of March to first Sunday of November. All Q2 and Q3 meetings fall within CDT. Q1 and Q4 meetings may cross DST boundaries -- handle per-meeting.

**Step 7: HTML Page Updates (WORK-03 part 3)**
- Read each HTML page, identify the relevant section by looking at adjacent existing entries, and write targeted updates following the same markup pattern. Do NOT rewrite entire HTML files.
- **newsletter.html**: Update "Current Issue" section to feature the new edition. Add the new edition to the appropriate year's archive section.
- **meetings.html**: Update the meeting schedule section with the new quarter's meetings. Update the "Download complete schedule" link to point to the new quarterly ICS file.
- **index.html**: Update the "Upcoming Meeting" section with the first active (non-cancelled) meeting of the new quarter. Update the "Read Latest Issue" / newsletter link to point to the new PDF. Update highlights section.
- **about.html**: Compare extracted officer roster against current about.html content. Update ONLY if there are changes to officers/BOG members.
- **contact.html**: Compare extracted mailing address against current contact.html. Update ONLY if there are changes.

**Step 8: ESBuild Rebuild (WORK-02)**

Failure to rebuild was the root cause of the "stale meetings" bug in v1.2 (documented in `.planning/debug/meetings-stale-content.md`).
- Verify the build succeeds (exit code 0). If it fails, report the error and stop.
